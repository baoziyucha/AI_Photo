# 摄影师与模特经验固化方案

> 面向：AI 随身摄影师当前项目落地  
> 参与角色：资深专业摄影师 & 资深专业模特 / 高级产品经理 / 高级全栈工程师  
> 目标：把专业人像摄影与模特引导经验，从“知识点”固化为当前 Android + Python 项目可实现、可测试、可演示的产品能力。

---

## 1. 三方交互结论

### 1.1 最终共识

AI 随身摄影师要固化的不是零散摄影知识，而是一条现场导演链路：

```text
检测事实
  -> 判断最关键问题
  -> 给一个动作指令
  -> 用语音带节奏
  -> READY 捕捉决定性瞬间
  -> 连拍选片
  -> 拍后复盘
  -> 反哺下一次指导
```

用户感知上，它不应该像“检测报告”，而应该像一个会说话的摄影师：一次只指出最影响出片的一件事，并把建议说成可执行动作。

典型体验不是：

```text
主体偏左，亮度异常，表情异常，角度异常。
```

而是：

```text
肩膀侧一点，脸转向光，好，睁眼，看镜头，拍。
```

### 1.2 关键取舍

1. P0 不追求完整专业摄影体系，先做高频、可检测、低争议的人像问题：眨眼、三轴头姿、表情自然度、左右脸光比、构图、主体位置。
2. 实时端只输出最高优先级的一条建议，避免用户在拍摄中被多条指令打断。
3. Kotlin 端负责实时硬判断和断网可用：READY、拍照拦截、TTS 指令、方向箭头、目标框、曝光补偿。
4. Python 服务端负责软点评和复杂语义增强：背景是否乱、氛围是否好、姿态是否自然、下一张怎么拍。
5. P0 暂不追求完整 Pose Detection。先用 ML Kit Face 的 `headEulerX/Y/Z`、脸框、睁眼、笑容和左右脸亮度，把头脸表情、光线和姿态做到足够专业。
6. 文案必须温和、具体、少废话、多鼓励。避免“脸大、显胖、难看”等羞辱感表达，改成“更自然、更立体、更舒展”。

### 1.3 必须写入 P0 的能力

- 接入 `headEulerX/Z`，统一三轴头姿语义，逐步废弃含义混乱的 `facePitchDeg`。
- 新增左右脸光比，识别阴阳脸、脸朝暗面、侧逆光。
- 新增结构化 `GuidanceIssue`，把规则命中结果、优先级、文案、TTS、READY 影响统一起来。
- 新增人物动作指令，不只指导“镜头向左”，也指导“下巴收一点、脸转向光、肩膀侧一点”。
- 新增 TTS 抓拍剧本 v1，例如“闭眼、深呼吸、3、2、1、睁眼看镜头”。
- `PhotoScorer` 增加表情、头姿、左右脸光比、基础姿态废片项。
- `server/prompts.py` 从“主体居中”改为“人像摄影语法”，输出构图、光线、表情、姿态、背景、镜头距离分维度建议。

## 2. 当前项目承接点

### 2.1 已具备的骨架

当前项目已经不是空白项目，已有能力可以直接承接经验固化：

| 模块 | 当前能力 | 可承接的经验 |
|---|---|---|
| `DetectionPipeline.kt` | ML Kit Face、主体框、亮度、倾斜、笑容、睁眼、`headEulerY` | 表情、眨眼、视线留白、主体位置 |
| `GuidanceEngine.kt` | 实时建议、方向箭头、目标框、READY 评分 | 摄影规则、动作建议、拍摄时机 |
| `GuidanceStabilizer.kt` | 连续帧稳定、防抖、保留稳定指导 | 语音节流、READY 稳定窗口 |
| `PhotoScorer.kt` | 拍后评分、构图/光线/倾斜/脸角度 | 拍后复盘、废片原因、下一张建议 |
| `GuidanceSpeaker.kt` | TTS 播报 READY、方向、光线 | 语音导演、抓拍剧本 |
| `CloudGuidanceClient.kt` | 云端实时增强、采样、去重、超时兜底 | 复杂场景软点评 |
| `server/prompts.py` | 实时指导 prompt、拍后点评 prompt | 云端人像摄影语法 |

### 2.2 已落地的人像专业能力

- 三分法与视线留白。
- 头顶留白。
- 眨眼拦截。
- 真笑、假笑、尴尬笑初判。
- 负空间拍后识别。
- READY 综合评分。
- 方向箭头、目标框、TTS 提示。

### 2.3 最大缺口

1. 数据层不完整：已采集 `smile`、双眼睁眼概率、`headEulerY`，但缺 `headEulerX/Z`。
2. 规则层仍偏硬编码：摄影经验散落在 `GuidanceEngine` 的 if/else 和文案分支里。
3. 光线专业度不足：当前只粗分全局亮度、左右/上下光源，缺左右脸光比、阴阳脸、眼神光、硬光/柔光。
4. 拍后评分缺人像废片维度：缺姿态、肩颈/关节裁切、背景干扰、手部遮挡、肤色/色温。
5. 云端 prompt 仍有“居中”惯性，和端上三分法/视线留白目标可能冲突。

## 3. 经验固化模型

建议把专业摄影师与模特经验拆成 6 类资产，每类都能映射到当前项目。

| 资产类型 | 内容 | 当前项目落点 |
|---|---|---|
| 检测事实 | 人脸框、主体框、笑容、睁眼、三轴头姿、主体面积、光源方向、左右脸亮度、背景复杂度 | `GuidanceFrame`、检测摘要 |
| 摄影规则 | 三分法、视线留白、头顶留白、阴阳脸、眨眼、假笑、裁关节、广角怼脸 | `GuidanceIssue`、规则引擎 |
| 动作指令 | 摄影师移动、被拍者动作、相机设置、保持/抓拍 | `CameraAction` + 新增人物动作 |
| 话术库 | 纠错句、鼓励句、抓拍口令、拍后复盘句 | `GuidanceSpeaker`、server prompt |
| 评分维度 | 构图、表情、姿态、光线、镜头距离、背景、成片情绪 | `PhotoScorer`、云端点评 |
| 拍摄剧本 | 闭眼深呼吸、八分脸、走动回头、窗边侧光、头像照 | TTS 状态机 + 连拍选片 |

### 3.1 单条经验规则结构

每条经验建议统一为如下结构，便于测试、排序、复用和云端同步：

```text
规则 id：expression.natural_smile
规则名：自然笑容抓拍
适用场景：portrait / selfie
触发条件：smile > 0.7 且 eyeOpen 在 0.4-0.8
实时动作：READY，可触发连拍
语音话术：很好，这个笑很自然，保持
拍后加分：表情自然 +15
风险兜底：eyeOpen < 0.3 时优先判定眨眼，不触发 READY
```

## 4. 建议数据模型

### 4.1 GuidanceIssue

新增统一规则命中模型，让 UI、TTS、PhotoScorer、云端 prompt 消费同一组专业判断。

```kotlin
data class GuidanceIssue(
    val id: String,
    val category: GuidanceCategory,
    val severity: IssueSeverity,
    val confidence: Float,
    val priority: Int,
    val message: String,
    val ttsMessage: String? = null,
    val action: SubjectAction? = null,
    val blocksReady: Boolean = false,
    val scoreDelta: Int = 0,
    val retryReason: RetryReason? = null,
)
```

建议新增分类：

```kotlin
enum class GuidanceCategory {
    SAFETY,        // 眨眼、主体丢失、严重过曝、严重倾斜
    COMPOSITION,   // 三分法、视线留白、头顶留白
    LIGHT,         // 阴阳脸、逆光、眼神光、柔硬光
    EXPRESSION,    // 真笑、假笑、尴尬笑、未笑
    POSE,          // 下巴、头歪、脸转角度、肩膀
    FRAMING,       // 裁切、贴边、主体大小
    BACKGROUND,    // 背景杂乱、亮斑、线条穿头
    LENS,          // 广角怼脸、焦段建议
)
```

### 4.2 FaceLightMetrics

用于把“看光”的摄影经验固化为可计算事实。

```kotlin
data class FaceLightMetrics(
    val leftFaceLuma: Float? = null,
    val rightFaceLuma: Float? = null,
    val faceLightRatio: Float? = null,
    val shadowSide: FaceSide? = null,
    val hasCatchLight: Boolean? = null,
)
```

P0 先做左右脸光比：

```text
左右脸亮度差明显 -> light.split_face
文案：侧光有点强，脸再转向光源一点。
READY 影响：降级到 ALMOST_READY 或 NOT_READY，取决于差值。
```

### 4.3 PortraitPoseMetrics

用于把模特经验中的“头、下巴、脸、肩膀”先轻量固化。

```kotlin
data class PortraitPoseMetrics(
    val headEulerX: Float? = null, // 低头/抬头
    val headEulerY: Float? = null, // 左右转脸，已用于视线留白
    val headEulerZ: Float? = null, // 歪头
    val chinAdviceState: ChinAdviceState? = null,
    val faceAngleState: FaceAngleState? = null,
)
```

规则示例：

| 触发 | 摄影含义 | 动作话术 |
|---|---|---|
| `headEulerX` 过低 | 低头、眼神弱、可能双下巴 | 下巴微收，眼睛看镜头上方一点 |
| `headEulerX` 过高 | 抬头过多、鼻孔外露 | 下巴稍微收回来 |
| `abs(headEulerZ) > 6` | 头歪过度 | 头扶正一点 |
| `abs(headEulerY)` 在 15-30 | 八分脸 | 这个侧脸角度很好，保持 |
| `abs(headEulerY) > 35` | 转脸过多、眼神丢失 | 脸转回来一点，看得到眼神更好 |

### 4.4 BackgroundMetrics

P0 可只做文档设计，P1/P2 再落代码。

```kotlin
data class BackgroundMetrics(
    val clutterScore: Float? = null,
    val hasHotspotNearHead: Boolean? = null,
    val hasVerticalLineAboveHead: Boolean? = null,
    val subjectSeparationScore: Float? = null,
)
```

## 5. 实时规则优先级

实时建议必须像摄影师一样“一次只说一件事”。建议主建议选择顺序：

| 优先级 | 类别 | 例子 | 是否阻断 READY |
|---|---|---|---|
| P0 | 安全废片 | 眨眼、主体丢失、严重过曝、严重倾斜 | 是 |
| P1 | 光线废片 | 阴阳脸、严重逆光、脸太暗 | 通常是 |
| P2 | 构图废片 | 贴边、头顶被切、视线看墙 | 视程度 |
| P3 | 姿态废片 | 头歪、下巴过低/过高、脸转太多 | 视程度 |
| P4 | 表情微调 | 假笑、尴尬笑、没笑 | 通常否 |
| P5 | 鼓励与加分 | 光线很好、笑容自然、八分脸很好 | 否 |

选择规则：

```text
1. 从 diagnostics 中筛选 confidence 足够的 issue。
2. 按 priority、severity、blocksReady 排序。
3. 只选择第一条作为实时 recommendationText 和 TTS。
4. 其他 issue 保留给拍后复盘和详情。
```

## 6. 产品体验闭环

### 6.1 主流程

1. 选择场景：头像、人像半身、旅行打卡、自拍、职场照。
2. 实时观察：采集主体、脸、光、表情、头姿、倾斜。
3. 单动作指导：只给最高优先级建议。
4. READY 捕捉：构图、光线、表情、姿态、稳定都达标后进入 READY。
5. 动作剧本：TTS 带节奏，让被拍者自然进入动作。
6. 连拍选片：READY 或剧本关键节点触发短连拍。
7. 拍后复盘：解释这张为什么好，下一张怎么改。
8. 个性化沉淀：记录用户更适合的侧脸、表情、构图和常见问题。

### 6.2 场景优先级

P0 建议聚焦三个场景：

| 场景 | 为什么优先 | 第一批剧本 |
|---|---|---|
| 头像/半身人像 | 最容易演示专业差异 | 八分脸、下巴、眼神、自然笑 |
| 自拍 | TTS 价值最明显 | 手机举高、下巴、微笑、READY |
| 旅行打卡 | 用户高频场景 | 三分法、视线留白、背景避让 |

## 7. 风格固化但不趋同

### 7.1 问题定义

风格趋同的根源不是“规则太多”，而是规则背后只有一种审美：默认好看等于端正、明亮、微笑、三分法、显瘦。如果系统把所有人都往这个安全区修正，就会抹掉人物气质、场景意图和照片表达。

因此经验固化必须分层：

```text
硬规则：保证照片成立，避免废片
风格偏好：决定照片往哪个审美方向走
受控变化：避免同一问题总是同一句话、同一个动作
个性化偏好：记住用户喜欢和不喜欢的方向
```

硬规则不能被风格覆盖；风格偏好不能被说成“错误”。例如眨眼、严重欠曝、脸被切坏是硬问题；是否微笑、是否居中、是否看镜头、背景是否必须干净、阴影是否保留，则应该随风格变化。

### 7.2 硬规则与风格规则边界

| 类型 | 是否可被风格覆盖 | 例子 | 输出方式 |
|---|---|---|---|
| 硬规则 | 不可覆盖 | 严重欠曝、主体虚焦、眨眼、脸被挡、关键部位被裁坏 | “这会影响照片有效性，需要先处理” |
| 风格偏好 | 可调权重 | 微笑/不笑、居中/留白、柔光/硬光、背景干净/环境参与 | “如果想要这个风格，可以这样做” |
| 受控变化 | 只影响表达 | 同一问题换不同动作、不同语气、不同抓拍节奏 | “同一轮稳定，不同轮有新鲜感” |
| 个性化偏好 | 微调风格 | 喜欢自然笑、偏暖色、不喜欢夸张摆拍 | “按用户偏好调权重，但不覆盖硬规则” |

输出结构建议固定为：

```text
必改问题：客观质量或用途风险
可选路线：不同风格下可以怎么拍
保留亮点：当前照片或人物气质中值得保留的部分
```

这样 AI 不会把“冷感风不笑”误判为“表情不好”，也不会把“街拍环境杂乱”一律判成背景失败。

### 7.3 风格坐标，而不是单一模板

产品上不建议只提供几个死模板，而应使用风格坐标：

| 坐标 | 两端含义 |
|---|---|
| 自然 / 戏剧 | 生活感、真实瞬间 / 强光影、强情绪 |
| 干净 / 氛围 | 背景简洁、主体突出 / 环境参与叙事 |
| 商业 / 生活 | 稳定、可信、可交付 / 松弛、随机、现场感 |
| 克制 / 张扬 | 表情内收、姿态少 / 动作外放、冲击力强 |
| 真实 / 理想化 | 保留个人特质 / 更修饰、更精致 |

用户可以选择目标，也可以选择负向偏好，例如“不想太网红”“不要过度磨皮”“不要太甜”“不要证件照感”。系统也可以从样片或历史高分照片推断风格，但必须允许用户本次关闭历史偏好。

### 7.4 风格分类与引导差异

| 风格 | 构图/光线 | 表情/姿态 | 话术样例 |
|---|---|---|---|
| 清新 | 高明度、柔光、干净背景 | 自然轻笑、放松、有呼吸感 | 像刚看到熟人一样轻轻笑 |
| 职场 | 稳定构图、干净轮廓光 | 克制自信、肩颈挺拔 | 下巴微收，肩放松，眼神像在做决定 |
| 胶片 | 可接受颗粒、偏色、逆光、轻微不完美 | 生活化，不必端正 | 别看我，像在等一阵风过去 |
| 街拍 | 环境参与构图，硬光可接受 | 动作感、酷、松弛或无表情 | 继续走，眼睛看前面，不用理镜头 |
| 情绪 | 低明度、阴影、留白 | 内收、侧身、低头可保留 | 把注意力放到自己身上，像刚想起一件事 |
| 甜美 | 柔光、近景、浅景深 | 笑容明显，手部可整理头发 | 笑到眼睛里，但嘴角不用太用力 |
| 冷感 | 低饱和、硬线条、少笑 | 克制、直线感、眼神坚定 | 眼神穿过镜头，不要讨好镜头 |
| 旅行 | 人物与地点关系明确 | 开放、自然、有互动 | 先看风景，再回头看我 |
| 证件感 | 正脸、均匀光、背景简洁 | 中性友好、端正 | 坐高一点，嘴角放松到刚好不严肃 |

这张表的意义不是把用户锁进风格，而是让同一个检测问题在不同风格下有不同解释。例如“脸部偏暗”在证件照里是必须修正，在电影感里可能只需要保留眼神光，在街拍里则可以建议走到更有方向性的自然光边缘。

### 7.5 技术模型：StylePreset / StyleProfile / RuleVariant

建议新增三层风格模型：

```text
StylePreset：系统内置风格方向，只定义审美倾向，不直接写死文案
StyleProfile：运行时风格实例，由 preset + userPreference + sessionContext 合成
RuleVariant：同一规则在不同风格下的权重、严重度、动作和话术
```

Kotlin 数据结构可参考：

```kotlin
data class StyleProfile(
    val presetId: String,
    val weights: Map<GuidanceCategory, Float>,
    val allowedActions: Set<SubjectAction>,
    val promptLexicon: PromptLexicon,
    val speechTone: SpeechTone,
    val userBias: UserPreference,
)
```

`GuidanceEngine` 的职责应从“规则直接产出建议”改为：

```text
检测事实 -> 产生 GuidanceIssue -> RuleVariantResolver 根据 StyleProfile 调整权重/动作/话术 -> 选择主建议
```

### 7.6 同一规则的风格变体

| 规则 | 清透/证件 | 电影/情绪 | 街拍/纪实 |
|---|---|---|---|
| `light.face_dark` 脸部偏暗 | 补正面柔光，让脸更清楚 | 可保留低调光，但要有眼神光 | 走到有方向性的自然光边缘 |
| `composition.off_center` 主体不居中 | 证件/职场需回到稳定位置 | 可保留偏边，增强孤独感 | 让环境多说一点，不急着居中 |
| `expression.no_smile` 没笑 | 可给轻微友好表情 | 可保留冷静或情绪感 | 可保持酷一点，不用理镜头 |
| `background.clutter` 背景杂乱 | 换干净背景 | 若氛围成立可降低惩罚 | 环境是故事的一部分，只避开抢脸元素 |

### 7.7 防同质化生成约束

建议在本地规则和云端 prompt 都加入以下约束：

1. 同一用户连续建议不得重复相同句式和同一审美方向。
2. 同一 `GuidanceIssue` 可以有多个同义动作，但硬规则判断必须稳定。
3. 每次拍后复盘至少包含一个“保留点”，避免 AI 只做纠错。
4. 可选优化至少给两条风格路线，例如“更干净”和“更有氛围”。
5. 使用受控随机：`seed = sessionId + styleId + issueId`，保证同一轮拍摄口径稳定，不同轮有变化。
6. 维护禁用模板句库，限制高频空话，例如“笑一下”“往左一点”“背景干净点”。

### 7.8 对 Kotlin / Python / TTS / 评分的改造要求

| 模块 | 改造方向 |
|---|---|
| `GuidanceEngine` | 规则先产出 `GuidanceIssue`，再由 `StyleProfile` 映射成动作与话术 |
| `PhotoScorer` | 拆成基础分 + 风格分；基础分看清晰、曝光、主体完整，风格分看该风格下的光线、背景、姿态是否匹配 |
| `GuidanceSpeaker` | 每个风格维护不同语速、节奏、动作剧本；清透风短而温和，电影感慢一点，街拍更即时 |
| `server/prompts.py` | 注入 `style_profile`、`issue_list`、`variant_policy`、`forbidden_homogenization` |
| 用户偏好 | 记录喜欢/不喜欢的风格，不允许个性化覆盖硬规则 |

### 7.9 去同质化验收

| 测试 | 验收标准 |
|---|---|
| 规则一致性 | 同一张严重欠曝照片，在所有风格下都触发硬规则 |
| 风格差异 | 同一张边缘案例图，不同 `StyleProfile` 下建议动作、评分权重、TTS 有可解释差异 |
| 文本相似度 | 不同风格建议文本相似度低于设定阈值 |
| 动作分布 | top issue 可以相同，但 top action 不应长期相同 |
| 高频词监控 | 每个风格的高频词和动作分布有明显差异 |
| 人工评审 | 同图多风格输出应像“同一个摄影助理换了风格”，不是同一句建议换皮 |

## 9. 话术与动作剧本样例

### 9.1 错误状态到摄影师话术

| 状态 | 避免说法 | 推荐话术 |
|---|---|---|
| 眨眼 | 你闭眼了 | 你眨眼了，再来一张 |
| 假笑 | 笑得很假 | 放松点，想想开心的事，真笑一下 |
| 阴阳脸 | 光线很差 | 侧光有点强，脸再转向光源一点 |
| 头歪 | 头歪了 | 头扶正一点，会更精神 |
| 下巴过高 | 鼻孔露出来了 | 下巴稍微收回来一点 |
| 低头 | 有双下巴 | 眼睛看镜头上方一点，下巴轻轻收住 |
| 广角怼脸 | 脸变形 | 退一步，用 2x 拍，脸型会更自然 |
| 背景抢戏 | 背景太乱 | 背景有点抢注意，换个角度更干净 |

### 9.2 P0 抓拍剧本 v1：自然笑头像

```text
步骤 1：肩膀侧一点，身体不用正对镜头。
步骤 2：脸转回来一点，看镜头上方。
步骤 3：下巴轻轻收住，嘴角放松。
步骤 4：闭眼，深呼吸。
步骤 5：3、2、1，睁眼看镜头。
步骤 6：很好，保持，拍。
```

触发建议：

```text
进入剧本：用户点击“语音导演”或连续 ALMOST_READY 超过 2 秒。
READY 条件：未眨眼、表情自然、头姿达标、光线不过曝/不严重阴阳脸。
后续动作：READY 瞬间可连拍 3-5 张，后续由 PhotoScorer 选最佳。
```

### 9.3 P1 剧本：窗边侧光半身

```text
步骤 1：站到窗边，让光从侧前方照到脸。
步骤 2：身体侧 30 度，肩膀放松。
步骤 3：脸转向光一点，让眼睛里有光。
步骤 4：看窗外，再慢慢看回镜头。
步骤 5：保持这个角度，拍。
```

## 10. 云端 prompt 改造方向

### 10.1 实时指导 prompt 原则

实时云端指导必须遵守端上人像规则：

- 不默认要求主体居中。
- 三分法 + 视线留白是合法构图。
- 只输出一条最关键动作。
- 眨眼、严重曝光、主体丢失等端上硬规则优先。
- 如果本地 `diagnostics` 已经给出高置信度问题，云端不要给冲突建议。

建议请求体增加：

```json
{
  "scene_type": "portrait",
  "face_count": 1,
  "face_positions": [],
  "head_euler": {"x": 2.1, "y": -12.0, "z": 1.5},
  "expression": {"smile": 0.82, "left_eye_open": 0.55, "right_eye_open": 0.61},
  "face_light": {"ratio": 1.8, "shadow_side": "right"},
  "readiness": "almost_ready",
  "diagnostics": [
    {"id": "light.split_face", "severity": "medium", "message": "侧光有点强"}
  ]
}
```

### 10.2 拍后点评 prompt 原则

拍后点评不只输出“优点/问题/建议”，应按人像维度给出结构化结果：

```json
{
  "summary": "整体表情自然，构图有留白，但右脸略暗。",
  "scores": {
    "composition": 86,
    "light": 72,
    "expression": 90,
    "pose": 78,
    "background": 80
  },
  "strengths": ["三分法和视线留白到位", "笑容自然"],
  "issues": ["右脸偏暗", "头部略向右歪"],
  "next_actions": ["脸再转向光源一点", "头扶正一点再拍"]
}
```
