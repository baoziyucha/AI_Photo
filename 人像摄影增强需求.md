## 一、背景与目标

### 1.1 产品定位
产品核心是「AI 随身摄影师」,主要功能是**拍人像**。当前已具备:场景识别、主体检测、构图/亮度/倾斜评估、方向箭头引导、拍摄就绪信号、拍后评分、云端点评。

### 1.2 核心问题
经专业摄影师视角评审,当前产品在**人像摄影专业性**上存在三类硬伤:

1. **免费数据未用**:[FaceSubjectDetector](app/src/main/java/com/yuxiang/aiphoto/analysis/DetectionPipeline.kt#L36-L72) 已开启 `LANDMARK_MODE_ALL`,但 ML Kit 提供的笑容/眨眼/三轴头部角度等核心人像数据完全未使用。
2. **构图逻辑反专业**:强制主体居中([TARGET_CENTER_X = 0.5f](app/src/main/java/com/yuxiang/aiphoto/analysis/GuidanceEngine.kt#L67)),违反人像三分法 + 视线留白铁律,会把好人像纠正成证件照。
3. **关键维度缺失**:无表情/眨眼检测、无阴阳脸检测、无裁关节检测、无背景干扰检测——这些都是人像废片的高频原因。

### 1.3 目标
把「AI 随身摄影师」从「懂居中的规则引擎」升级为「懂人像的 AI 摄影师」,使其在 TRAE 大赛现场真机演示中,评委能直观感知「它真的懂摄影」。

### 1.4 成功指标(演示视角)
- 评委眨眼时,AI 主动提示「你眨眼了,重拍」
- 主体偏左看向右留白时,AI 不纠正位置,直接标 READY(展示懂三分法)
- 侧光阴阳脸时,AI 提示「侧光过强,转脸 30°」(展示懂光位)

## 二、需求清单

### P0-1 启用 ML Kit 已有的人像数据(零成本杀手锏)

**问题**:[FaceSubjectDetector](app/src/main/java/com/yuxiang/aiphoto/analysis/DetectionPipeline.kt#L36-L72) 当前只用了眼/嘴/鼻 4 个 landmark 估算一个角度,而 ML Kit `Face` 对象上还有大量未使用的人像核心数据。

**ML Kit 已提供、当前未用的字段**:

| ML Kit 字段 | 摄影意义 |
|---|---|
| `headEulerAngleX` | 抬头/低头(双下巴/鼻孔外露) |
| `headEulerAngleY` | 左右转头(**决定视线留白方向**) |
| `headEulerAngleZ` | 歪头(脖子倾斜) |
| `smilingProbability` | 笑容概率(**决定性瞬间**) |
| `leftEyeOpenProbability` / `rightEyeOpenProbability` | 眨眼(**废片元凶**) |

**需求**:
1. `FaceDetectionResult` 扩展字段:`headEulerX/Y/Z`、`smilingProbability`、`leftEyeOpenProb`、`rightEyeOpenProb`
2. `FaceSubjectDetector.detect()` 直接读取这些字段并返回(无额外计算成本)
3. 顺带修复命名 bug:当前 `facePitchDeg` 实际计算的是 **yaw**(左右转头),应改名 `faceYawDeg`,避免后续接入更多角度时混淆

**改动文件**:
- [DetectionPipeline.kt](app/src/main/java/com/yuxiang/aiphoto/analysis/DetectionPipeline.kt)(数据采集)
- [CameraGuidanceModels.kt](app/src/main/java/com/yuxiang/aiphoto/model/CameraGuidanceModels.kt)(数据模型)

**验收**:
- 单元测试覆盖:各角度/概率字段正确从 ML Kit 透传
- 真机日志可见 `smilingProbability` 等数值

**代码现状确认**(2026-06-26 核对):
- [DetectionPipeline.kt#L37-L44](app/src/main/java/com/yuxiang/aiphoto/analysis/DetectionPipeline.kt#L37-L44) 已开启 `LANDMARK_MODE_ALL`,但 [detect()](app/src/main/java/com/yuxiang/aiphoto/analysis/DetectionPipeline.kt#L46-L72) 仅用 landmark 估算单一角度,未读取 `headEulerAngleX/Y/Z`、`smilingProbability`、`leftEyeOpenProbability`、`rightEyeOpenProbability`
- **命名 bug 确认**:[estimateFacePitch()](app/src/main/java/com/yuxiang/aiphoto/analysis/DetectionPipeline.kt#L74-L98) 实际计算 `atan2(dx, dy)`(双眼与嘴/鼻的横向偏移),本质是 **yaw**(左右转头),却命名为 `facePitchDeg`。建议直接废弃自算逻辑,改用 ML Kit 的 `headEulerAngleY`(精度更高且语义明确)
- [FaceDetectionResult](app/src/main/java/com/yuxiang/aiphoto/analysis/DetectionPipeline.kt#L19-L24) 当前只有 `subjectBox/faceCount/confidence/facePitchDeg` 四字段
- [GuidanceFrame](app/src/main/java/com/yuxiang/aiphoto/model/CameraGuidanceModels.kt#L116-L151) 当前只有 `facePitchDeg: Float?`,无人像表情数据

**数据模型改动细节**:

`FaceDetectionResult` 扩展(改名同时修正语义):
```kotlin
data class FaceDetectionResult(
    val subjectBox: NormalizedRect?,
    val faceCount: Int,
    val confidence: Float,
    // 废弃 facePitchDeg,改用 ML Kit 三轴

    val headEulerY: Float? = null,   // yaw:摇头(+左/-右,决定视线留白)
    val headEulerZ: Float? = null,   // roll:歪头

```

`GuidanceFrame` 同步扩展(传递到 GuidanceEngine/PhotoScorer/UI):
```kotlin
data class GuidanceFrame(

    val facePitchDeg: Float? = null,  // 过渡期保留,后续废弃
```

**迁移策略**(渐进式,降低风险):

3. 最后删除 `facePitchDeg`

### P0-2 表情与眨眼检测(决定性瞬间)

**问题**:当前 [CaptureReadiness](app/src/main/java/com/yuxiang/aiphoto/analysis/GuidanceEngine.kt#L234-L267) 只看构图/亮度/倾斜/缩放,完全没看表情。一个不拦截眨眼的 AI 摄影师,在评委眼里不专业。

**需求**:
1. **眨眼拦截**:任一眼 `openProbability < 0.3` → 强制 `CaptureReadiness = NOT_READY`,文案「你眨眼了,再来一张」
2. **笑容提示**:
   - 自拍/人像场景,`smilingProbability < 0.2` 且持续 >2s → 提示「可以笑一下」
   - `smilingProbability > 0.7` → 表情分满分,纳入 READY 评分
3. **READY 评分新增「表情分」维度**:权重建议 0.15,从现有的构图/亮度/倾斜/缩放中按比例下调

**改动文件**:
- [GuidanceEngine.kt](app/src/main/java/com/yuxiang/aiphoto/analysis/GuidanceEngine.kt)(评分逻辑、文案)
- [CameraGuidanceModels.kt](app/src/main/java/com/yuxiang/aiphoto/model/CameraGuidanceModels.kt)(若新增枚举)

**验收**:
- 评委眨眼演示:AI 提示重拍,READY 信号不触发
- 评委微笑演示:READY 信号正常触发,文案无错误拦截

### P0-3 三分法 + 视线留白(替代强制居中)

**问题**:[GuidanceEngine.kt#L67](app/src/main/java/com/yuxiang/aiphoto/analysis/GuidanceEngine.kt#L67) `TARGET_CENTER_X = 0.5f`、[PhotoScorer.kt#L92](app/src/main/java/com/yuxiang/aiphoto/analysis/PhotoScorer.kt#L92) 评分区间 0.42~0.58 才算「主体居中」得分。这违反人像摄影三条铁律:
1. **三分法**:主体应放左/右 1/3 交点,不是正中
2. **视线留白**:人脸朝哪边,留白就在哪边
3. **头顶留白**:头部上方留 5-12%

当前逻辑会把「主体在左 1/3、看向右留白」的标准好人像**纠正成证件照**,是产品最严重的专业性硬伤。

**需求**:
1. 用 `headEulerAngleY`(来自 P0-1)判断视线方向:脸朝右 → 留白在右
2. 目标框中心 X 动态计算:`if (看向右) 0.33 else 0.67`(替代固定 0.5)
3. 评分逻辑调整:主体在三分点 + 视线朝向留白 = 满分;只有当视线方向与留白方向矛盾(看向画面外)时才扣分
4. 保留「正中」作为合法构图之一(证件照/对称构图场景),不再作为唯一答案

**改动文件**:
- [GuidanceEngine.kt](app/src/main/java/com/yuxiang/aiphoto/analysis/GuidanceEngine.kt)(`computeTargetZone`、`evaluateComposition`)
- [PhotoScorer.kt](app/src/main/java/com/yuxiang/aiphoto/analysis/PhotoScorer.kt)(居中评分改为三分法评分)

**验收**:
- 主体偏左 1/3、看向右留白 → READY,无「镜头向右」误导
- 主体偏左 1/3、看向左(看墙)→ 提示「让脸转向画面深处」
- 单元测试覆盖三分点 + 视线方向的组合

### P1-1 脸部受光均匀度(阴阳脸检测)

**问题**:[BrightnessEvaluator](app/src/main/java/com/yuxiang/aiphoto/analysis/GuidanceEngine.kt#L289-L296) 只看全图平均亮度。人像摄影里「阴阳脸」(半边亮半边暗)比「整体偏暗」更常见、更毁片,当前完全检测不到。

**需求**:
1. 取 ML Kit 脸部 boundingBox,分别统计**左半脸 / 右半脸**平均亮度
2. 差值 > 1.5EV → 文案「侧光过强,让脸转向光源 30°,或退到阴影里要柔光」
3. 该维度纳入亮度评分:阴阳脸时亮度分降级

**改动文件**:
- `BrightnessEvaluator`(新增左右脸分别统计)
- [GuidanceEngine.kt](app/src/main/java/com/yuxiang/aiphoto/analysis/GuidanceEngine.kt)(文案与评分)

**验收**:
- 侧光场景演示:AI 提示转脸或退到阴影,文案含「30°」「柔光」等专业措辞

### P1-2 肩颈裁切检测

**问题**:[PhotoScorer.kt#L106-110](app/src/main/java/com/yuxiang/aiphoto/analysis/PhotoScorer.kt#L106-L110) 只看「是否贴边」,但主体可能没贴边却正好被裁在膝盖/脖子。人像摄影基础常识:**不裁关节、不裁脖、不裁腕**。

**需求(分两档)**:

| 档位 | 方案 | 覆盖范围 |
|---|---|---|
| 轻量(先做) | 用 ML Kit Face 下巴 landmark + boundingBox 推断头部位置,若 bbox 下沿紧贴画面下沿 → 「镜头抬高一点,别切到肩膀」 | 头肩像 |
| 完整(后置) | 接 ML Kit Pose Detection,拿肩/肘/腕/髋/膝,关键点距边缘 <5% 时按部位报「别切到膝盖/手腕」 | 半身/全身像 |

**改动文件**:
- [PhotoScorer.kt](app/src/main/java/com/yuxiang/aiphoto/analysis/PhotoScorer.kt)
- 轻量档:`DetectionPipeline.kt` 增加 landmark 输出
- 完整档:新增 `PoseDetector`

**验收**:
- 头像裁到脖子 → AI 提示抬高镜头
- (完整档)半身像裁到膝盖 → AI 提示别切关节

### P1-3 姿态引导(抬头/歪头/双肩)

**问题**:[getFacePitchAdvice](app/src/main/java/com/yuxiang/aiphoto/analysis/GuidanceEngine.kt#L402-L413) 当前只会说「脸转正一点」。专业人像姿态引导应基于三轴角度 + 双肩。

**需求**:

| 角度 | ML Kit 字段 | 触发阈值 | 文案 |
|---|---|---|---|
| 低头/抬头 | `headEulerAngleX` | X > 10° | 「下巴抬一点,避免双下巴」 |
|  |  | X < -10° | 「头低了一点,抬眼看镜头」 |
| 歪头 | `headEulerAngleZ` | \|Z\| > 6° | 「头有点歪,扶正」 |
| 转头 | `headEulerAngleY` | \|Y\| > 25° | 「脸转太多,看不见眼神光了」 |
| 双肩 | (Pose 或脸部 bbox 宽高比推断) | 正对镜头 | 「肩膀侧一点,别正对镜头,更显瘦」 |

**改动文件**:
- [GuidanceEngine.kt](app/src/main/java/com/yuxiang/aiphoto/analysis/GuidanceEngine.kt)(`getFacePitchAdvice` 重构为三轴姿态引导)

**验收**:
- 演示「双肩侧一点显瘦」这一句——写真摄影师口头禅,瞬间提升专业感

### P2-1 背景干扰检测

**问题**:当前无背景评估。人像废片里「头顶长电线杆」「背景有抢眼高光」与构图问题一样高频。

**需求(轻量方案,无需新模型)**:
1. 取脸部 bbox **外围环带**(脸外扩 1.5 倍区域)的亮度方差和最大亮度
2. 出现局部高亮(>220)且面积 >5% → 「背景有高光抢戏,换个角度」
3. 脸上方有垂直亮线 → 「头顶有电线杆/柱子,挪一步」

**改动文件**:
- 新增 `BackgroundClutterEvaluator`
- [DetectionPipeline.kt](app/src/main/java/com/yuxiang/aiphoto/analysis/DetectionPipeline.kt) 接入

**验收**:
- 背景有亮灯/柱子时,AI 主动提示避让

### P2-2 黄金光位识别(主动型 AI)

## 四、演示场景建议(TRAE 大赛现场)

建议预埋 3 个「10 秒打动评委」的瞬间,覆盖人像三要素(瞬间/构图/光线):

| 场景 | 触发 | AI 反应 | 展示点 |
|---|---|---|---|
| 眨眼拦截 | 评委眨眼 | 「你眨眼了,再来一张」+ READY 不触发 | 瞬间 |
| 视线留白 | 主体偏左看向右 | 不纠正位置,直接 READY | 构图 |
| 阴阳脸 | 侧光 | 「侧光过强,转脸 30°」 | 光线 |

三个场景都能用现有 ML Kit 数据实现,无需新模型,无网络依赖,演示可靠性高。

## 五、风险与依赖

| 风险 | 说明 | 缓解 |
|---|---|---|
| ML Kit 字段缺失 | 部分老旧设备/低分辨率帧可能拿不到 `smilingProbability` | 字段为 null 时降级为「不评估表情」,不影响其他维度 |
| 视线方向误判 | `headEulerAngleY` 在侧脸时精度下降 | 设置置信度阈值,低置信度时回退到「居中 + 头顶留白」保守策略 |
| 三分法评分抖动 | 主体在三分点附近来回时 READY 抖动 | 复用现有 [GuidanceStabilizer](app/src/main/java/com/yuxiang/aiphoto/analysis/GuidanceStabilizer.kt),连续 N 帧一致才输出 |
| Pose Detection 体积 | P1-2 完整档引入 ML Kit Pose 会增加 APK 体积 | 先做轻量档(纯 ML Kit Face landmark),完整档后置评估 |

## 六、明确不做(scope 边界)

为聚焦人像核心,以下本次**不做**:
- 风景 / 美食 / 夜景 / 建筑等非人像场景分类
- 风景地平线检测(人像场景的地平线切脖问题由 P1-2 覆盖)
- 滤镜 / 色温 / 色彩和谐度建议
- 拍后智能裁剪建议
- 多人队形构图(三角形构图、视线一致性)

## 九、TRAE 大赛现场演示脚本(2026-06-26 补全)

> 灵感库第八章建议:预埋 3 个「10 秒打动评委」瞬间,覆盖人像三要素(瞬间/构图/光线)+ 1 个对比图杀手锏。

### 9.1 演示流程(建议 5 分钟)

| 时长 | 环节 | 操作 | AI 反应 | 展示点 |
|---|---|---|---|---|
| 0:00-0:30 | 开场 | 介绍「AI 随身摄影师」定位 | 取景框实时叠加箭头/目标框/READY 信号 | 实时性 |
| 0:30-1:30 | 瞬间①:眨眼拦截 | 评委眨眼 | 「你眨眼了,再来一张」+ READY 不触发 | **瞬间** |
| 1:30-2:30 | 瞬间②:视线留白 | 主体偏左 1/3、看向右 | 不纠正位置,直接 READY + 绿框脉冲 | **构图** |
| 2:30-3:30 | 瞬间③:阴阳脸 | 侧光场景 | 「侧光过强,转脸 30°」+ 转脸后 READY | **光线** |
| 3:30-4:30 | 瞬间④:抓拍剧本 | 触发抓拍模式 | TTS「闭眼深呼吸,3、2、1,睁眼」→ 连拍 5 张 → 推荐「这张最好」 | 流程 |
| 4:30-5:00 | 收尾 | 八分脸演示 | 侧脸 20° → 「这个侧脸角度很显瘦」 | 专业感 |

### 9.2 对比图杀手锏(灵感库 8.7,最高演示价值)

建议预录或现场演示「AI 指导前 vs 指导后」对比:
- **指导前**(关闭 AI):普通构图,主体居中,正脸,平光 → 证件照感
- **指导后**(开启 AI):三分法 + 视线留白,八分脸,前侧光,真笑 → 写真感

两张并排展示,效果差异肉眼可见。这是评委最容易记住的演示形式,比任何功能介绍都有效。

### 9.3 演示可靠性保障
- 所有演示瞬间均用现有 ML Kit 数据实现,无需新模型、无网络依赖
- 断网情况下本地指导完整可用([plan1.md](plan1.md) P0-4 云端兜底已落地)
- 现场光线不可控时,阴阳脸演示可改为「窗边侧光」场景(室内可复现)

### 9.4 风险预案
| 风险 | 预案 |
|---|---|
| ML Kit 字段缺失(老旧设备) | 字段 null 时降级为「不评估表情」,不影响其他维度(需求文档第五节已列) |
| READY 抖动 | 复用 [GuidanceStabilizer](app/src/main/java/com/yuxiang/aiphoto/analysis/GuidanceStabilizer.kt),连续 N 帧一致才输出 |
| 抓拍剧本 TTS 时机不准 | 「睁眼」检测设置置信度阈值,误判时手动触发连拍兜底 |
| 现场无侧光 | 改用窗边自然光,或用手机手电筒侧打光演示阴阳脸 |
